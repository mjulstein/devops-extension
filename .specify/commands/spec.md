# `/spec`

Use this command when a feature request or idea is clear enough to promote into a numbered spec under [`specs`](../../specs/README.md).

## Goals

- choose the next numbered spec folder
- translate user stories into prioritized scenarios and requirements
- keep planning artifacts aligned with the repository constitution and documentation rules

## Command Checklist

1. Review [`.specify/memory/constitution.md`](../memory/constitution.md).
2. Review [`specs/README.md`](../../specs/README.md) for promotion flow and active specs.
3. If applicable, review the source idea note in [`specs/ideas`](../../specs/ideas/README.md).
4. Start from [`.specify/templates/spec-template.md`](../templates/spec-template.md).
5. Create `specs/###-feature-name/spec.md` with prioritized user stories, acceptance scenarios, edge cases, requirements, entities, and success criteria.
6. Update related planning indexes or navigation docs when the new numbered spec would otherwise be hard to discover.

## Expected Output

A numbered feature spec that is specific enough to guide later planning and implementation without prematurely committing to code-level details.
